**Мета**: Опанувати Data Dive (Niche + RankRadar) як інструмент глибокого нішевого аналізу і трекінгу органічних позицій, привʼязати до Модуля 6 ranking reminder / ToS-чеків, і розібрати decision tree «який інструмент для якого питання». Лайфхаки з оригінального модуля — переосмислені як «use at own risk».

> [!note] Data Dive — платний, корпоративний доступ через керівника. У більшості акаунтів активовані Niche + RankRadar по топ-ASIN. Без активації конкретної ніші для акаунту ти в Data Dive нічого не побачиш — спершу дай аккаунт-менеджеру список ASIN/ключів для налаштування.

## 1. Коли підключається Data Dive

Data Dive — третій рівень діагностики (Seller Sprite = 1-й, AsinSight = 2-й, Data Dive = 3-й). Підключається коли:
- Треба точний і довготривалий трекінг власних органічних позицій → **RankRadar**.
- Треба глибокий огляд ніші з Master Keyword List і типізацією ключів (Outliers / Roots / Normalizer) → **Niche**.
- Готуєш Module 6 ranking reminder — перевірку, які ключі рухаються і куди.
- Перевіряєш гіпотези по structural change у ніші (доповнення до AsinSight).

**Порівняння з іншими інструментами:**

| Задача | Кращий інструмент |
|--------|-------------------|
| Швидкий SERP-check | Seller Sprite SERP-layer |
| 30-денні зміни на конкретному лістингу | Seller Sprite listing-overlay |
| Keyword gap між твоїм ASIN і 3 конкурентами | AsinSight Multi-ASIN |
| Хто тримає ToS на ключі + % часу утримання | AsinSight Keyword Research |
| Довготривалий трекінг органічних позицій власних ASIN | **Data Dive RankRadar** |
| Повна карта ніші (категорія / MKL / Outliers / Roots) | **Data Dive Niche** |
| Margin / Net Profit / BE-ACoS | Sellerboard (тільки він) |

## 2. Niche — аналіз ніші

**Шлях:** Data Dive → Niche → вибрати нішу (додається аккаунт-менеджером за ASIN або ключем).

**Основні вкладки:**

**Master Keyword List (MKL)** — головні ключі ніші, відсортовані за SV / релевантністю. Це стартовий набір для будь-якої нової кампанії в цій ніші. Використання: при запуску нового ASIN, при ревізії існуючих кампаній (чи не пропустили ключа з MKL).

**Outliers** — широкі ключі з високим SV, але низькою релевантністю. Приклад для `spice jars`: запит `kitchen storage` має SV 80,000 але релевантність 12%. Не варто додавати в exact — пожже бюджет без конверсії. Можливий варіант — як broad з моніторингом.

**Roots** — кореневі слова, з яких формуються broad / phrase-ключі. Для `spice jars` → `spice`, `jar`, `spice storage`. Корисно для конструювання phrase-кампаній і розуміння, де перетини з сусідніми нішами.

**Normalizer** — уникання дублів ключів. Ключі `garlic press kitchen` і `garlic press for kitchens` — Amazon вважає їх одним матчем. Normalizer показує такі пари, допомагає не засмічувати кампанії.

### Коли використовувати Niche

- **При запуску нового ASIN** — зібрати стартовий pack ключів (MKL → exact + phrase, Roots → broad).
- **Раз на 1–2 місяці по топ-ASIN** — перевірити, чи ніша не розширилась (нові ключі в MKL, які ми пропустили).
- **При плануванні Module 6 тригера #17** (прибутковий ASIN без SB/SBV/SD) — Roots дають матеріал для Sponsored Brands keyword targeting.

## 3. RankRadar — трекінг власних органічних позицій

**Шлях:** Data Dive → RankRadar → додати ASIN і ключі (ліміти узгоджує аккаунт-менеджер).

**Що показує:**
- Органічні позиції ASIN за кожним трекнутим ключем — з історією за 7 / 30 / 90 днів.
- Зміни позицій (з ↑ / ↓ індикатором).
- Додатково — SP-позиції (але для SP точніший AsinSight).
- Варіаційні ASIN — дані включають усі варіації parent ASIN (це треба враховувати, перевіряти вручну).

### Прив'язка до Module 6 ranking reminder

Ranking reminder — це правило: **якщо ASIN за 30 днів суттєво впав за органічним ranking на головному ключі, це пріоритетний сигнал, навіть якщо PPC-метрики поки нормальні**. RankRadar — головне джерело цього сигналу.

**Порядок використання:**
1. Щотижня (або раз на 2 тижні) — відкрий RankRadar по топ-10 ASIN портфоліо.
2. Сортуй за `Change WoW` або `Change 30d`.
3. ASIN з просадкою >5 позицій за 30 днів на головному ключі → **pre-trigger**, додавай у моніторинг-лист.
4. Якщо ця просадка підтверджується в Sellerboard Sessions drop (тригер #7) → повна діагностика.

### Прив'язка до ToS-чеків

ToS (Top of Search) це не просто рекламна позиція — це найдорожчий і найконверсійніший трафік. Data Dive дозволяє перевіряти, які твої ключі утримують ToS у SP і чи це стабільно:
1. У RankRadar або Niche дивись SP-позиції по трекнутих ключах.
2. Якщо ASIN тримає ToS > 60% часу на ключі з високим SV — це «золотий» ключ, який треба захищати (збільшений бюджет на exact, моніторинг CPC).
3. Якщо ASIN «тече» між ToS і Rest of Search — нестабільна ставка / високий варіативний CPC. Варіант: лог в bulk-оптимізацію → розглянути `Placement modifier` для ToS (див. Модуль 5).

### Коли використовувати RankRadar

- Тижнево по топ-ASIN як частина Skill 1 workflow (опціонально, але радикально підвищує ранню діагностику).
- При підозрі на Google-update / Amazon algo change (коли багато ASIN просіли синхронно).
- При запуску нового ASIN — RankRadar стає головною метрикою поки накопичуються перші продажі і статистика ACoS ще не репрезентативна.

## 4. Decision tree — який інструмент для якого питання

Коли виникає питання, йди по цьому дереву:

```
Питання: "Чому ASIN просів (Sessions / CR / Profit)?"
│
├── Fast check: Sellerboard Товари (Margin / Sessions / CR / Real ACoS)
│   └── Якщо метрики просіли → далі
│
├── Context 1: Seller Sprite SERP-check по головному ключу (60 сек)
│   └── Якщо позиція впала → Seller Sprite listing-overlay (конкуренти, ціни)
│
├── Context 2: AsinSight Keyword Research (якщо питання "хто зайшов у ToS")
│
└── Context 3: Data Dive RankRadar (якщо питання "коли саме почалась просадка ранків")
```

```
Питання: "Які ключі я пропустив?"
│
├── Швидко: AsinSight Multi-ASIN Analysis (gap аналіз)
│
└── Глибоко: Data Dive Niche → Master Keyword List
```

```
Питання: "Що з моїм profit?"
│
└── Sellerboard Товари + PPC Dashboard по портфоліо
    (жоден інший інструмент цього НЕ дає)
```

```
Питання: "Запускаємо новий ASIN — де брати ключі?"
│
├── Data Dive Niche → MKL (стартовий набір)
├── AsinSight Asin Research по топ-3 конкурентам (що у них працює)
└── Крос-перевірка: Helium 10 / Data Dive Outliers (щоб не взяти нерелевантне)
```

## 5. Лайфхаки — use at own risk

Оригінальний модуль подавав ці як «рекомендації». Насправді кожен має ризик — описую чесно:

### Temp Mail для безкоштовного AsinSight (30 днів × N)

**Що це:** реєструвати AsinSight з тимчасовими email для обходу лімітів безкоштовного періоду.
**Ризики:** порушення ToS AsinSight → бан IP / браузера / виявлення через browser fingerprint. Для акаунту-менеджера це ще й репутаційний ризик.
**Рекомендація:** тільки для персонального навчання, не в продакшн-роботі по 30 клієнтах. Для регулярного використання — платна підписка через компанію.

### Аналіз варіацій у AsinSight через parent ASIN

**Що це:** ввести parent ASIN → побачити розподіл трафіку по варіаціям (colors).
**Ризик:** низький. Корисно, щоб помітити «сліпі» варіації з <10% трафіку.
**Рекомендація:** використовуй, але перевіряй ручну — AsinSight не ідеально розпізнає мерджі variations.

### Швидкий аналіз утримань у Sellerboard

**Що це:** у Dashboard → Утримання → відняти FBA Fee + Referral Fee від загальної суми → отримаєш «additional» (повернення, зберігання, інші).
**Ризик:** низький. Корисно для швидкої оцінки «а скільки ми втратили на поверненнях».
**Рекомендація:** валідна техніка. Але якщо сума additional > 15% від Utility — потрібен детальний розбір з аккаунт-менеджером, а не просто «помітка».

### Використання таблиць маржинальності від акаунт-менеджерів

**Що це:** не рахувати Margin щоразу — взяти готову таблицю.
**Ризик:** високий — таблиці часто застарівають (COG не оновлюється, Amazon fees змінюються). Використання застарілої таблиці для target ACoS = оптимізація по кривій.
**Рекомендація:** використовуй ТІЛЬКИ як референс; target ACoS завжди рахуй з свіжого Sellerboard Товари.

## 6. Підсумок Модуля 7

Ти прошов повний стек додаткових інструментів:

| Крок | Інструмент | Головна задача |
|------|-----------|----------------|
| 1 | Seller Sprite SERP | 60-секундний діагноз «що з видачею» |

**Головне правило:** інструмент вмикається за конкретним питанням, а не «порозглядати для порядку». Твій час на 30 акаунтах обмежений — decision tree вище економить години.

### Практичний кейс

**Сценарій.** Аккаунт-менеджер питає: «У портфоліо `B07SPICE` Sessions впали синхронно по 4 з 5 ASIN за останній тиждень. ACoS зріс з 17% до 24%. Margin 22%. Дай аналіз за 20 хвилин і рекомендацію».

**Інструменти під рукою:** Sellerboard, Seller Sprite, AsinSight, Data Dive.

**Питання:**
1. Який decision flow ти вибираєш?
2. Який послідовність кроків за 20 хвилин?
3. Як формулюєш рекомендацію?

*Розв'язок:*

1. **Decision flow:** це синхронна просадка по портфоліо → виглядає як structural change у ніші (тригер #22). Спершу встановити, чи це PPC-явище чи нішеве.

2. **Послідовність (20 хв):**
   - **2 хв:** Sellerboard Товари — підтвердити метрики (Real ACoS, USP, Margin, Refund rate по 5 ASIN).
   - **5 хв:** Seller Sprite SERP по 3 головних ключах портфоліо — чи змінився розклад у топ-5.
   - **5 хв:** AsinSight Keyword Research по тих самих ключах — подивитись % утримання ToS за 7 днів, знайти нових рекламодавців.
   - **5 хв:** Data Dive RankRadar по 5 ASIN на головних ключах — підтвердити, що органічні позиції теж просіли (ключове питання: це тільки SP чи комплексний дроп).
   - **3 хв:** Seller Sprite listing-overlay на 2 топ-конкурентів — чи були зміни цін / картинок.

3. **Рекомендація** (приклад виводу):
   «Портфоліо `B07SPICE`: синхронна просадка Sessions −14 до −22% WoW по 4/5 ASIN. Причина комбінована:
   - AsinSight KW Research: за ключами `X`, `Y` зайшов новий бренд B08NEW, тримає ToS 32% часу.
   - Seller Sprite: B08NEW запустив ціну на 18% нижче середнього по ніші.
   - RankRadar: наші органічні ранки стабільні (−1 до −2 позиції — не критично) — просадка в основному на SP-стороні.

   **Дія:**
   - PPC: не підвищувати біди (ACoS вже +7 pp vs target 22%); перевірити доступність `Phrase/Broad` для харвесту трафіку поза ToS; розглянути SD-таргет на B08NEW.
   - Ескалація: аккаунт-менеджер — перевірити цінову позицію портфоліо.
   - Моніторинг: повторний RankRadar + AsinSight через 7 днів; якщо B08NEW закріпиться — переглянути структуру бідів під новий ціновий нормал ніші».

### Розрахункова задача

Дано RankRadar для ASIN `B07TEST` по ключу `ice cream scoop` за 30 днів:

| День | Organic Pos | SP Pos | Утримання ToS (%) |
|------|-------------|--------|--------------------|
| Day 1 (30 днів тому) | 4 | 2 | 22 |
| Day 15 | 5 | 2 | 18 |
| Day 30 (сьогодні) | 9 | 4 | 7 |

Sellerboard Товари: Margin = 18%, Sessions день 1 ~ 520, Sessions день 30 ~ 360.
PPC Kw Ex кампанія на `ice cream scoop`: Spend $240/тиждень (сьогодні), Sales $720, ACoS ?

**Порахуйте:**
1. % просадки Sessions за 30 днів.
2. % просадки утримання ToS за 30 днів (pp і відносно).
3. ACoS кампанії сьогодні.
4. target ACoS (BE-ACoS по портфоліо = 12%).
5. Який тригер Skill 3 / Module 6 спрацьовує і яка оптимальна дія?

*Розв'язок:*
1. Sessions WoW-тренд: (360 − 520) / 520 × 100 = **−30.8%** за 30 днів → стабільний дроп, тригер #7 (Sessions drop) спрацьовує.
2. % утримання ToS: 22 → 7 = **−15 pp**, відносно: (7 − 22) / 22 × 100 = **−68%**. Це дуже суттєва втрата ToS-часу.
3. ACoS = 240 / 720 × 100 = **33.3%**.
4. target = max(Margin 18, BE 12) = **18%**. Cap 30 не спрацьовує, floor 10 не потрібен.
5. Тригери: **#7 (Sessions drop)** + **#20 (ACoS > target, ACoS +15 pp, 1.85× target)** + **Module 6 #22 (competition pressure — втрата ToS)**. Рекомендація:
   - Не підвищувати біди — ACoS уже 1.85× target.
   - Перевірити через AsinSight, хто зайшов у ToS за останні 30 днів — можливо новий бренд / нова низька ціна.
   - Розглянути перерозподіл бюджету: частина у Phrase / Broad для харвесту кліків поза ToS.
   - Якщо через тиждень ToS не відновиться — знижувати біди на −15% і писати рекомендацію аккаунт-менеджеру щодо ціни / лістингу.
