**Мета**: Опанувати Sellerboard як source of truth для всього Skill 1 workflow — структуру Товари (Margin, COG, Real ACoS, Sessions, USP), PPC Dashboard по портфоліо (звідки береться Break-even ACoS для Модуля 5 Кроку 1) і Stock Planner.

> [!warning] Sellerboard — це не «ще один інструмент». Це **базовий файл** твого тижневого процесу. Без правильно вигружених Товари і PPC Dashboard по портфоліо ти фізично не можеш рахувати target ACoS, а отже не можеш коректно оптимізувати жодну кампанію. Усі тригери Skill 3 прив'язані до колонок саме з цих файлів.

## 1. Навіщо PPC-менеджеру Sellerboard

Seller Central + Campaign Manager дають тобі тільки Spend / Sales / ACoS у PPC-інтерфейсі — без реального профіту, без урахування комісій, FBA, повернень, COG. Тобто ти бачиш «рекламний ACoS», але не знаєш, чи приносить кампанія гроші на рівні бізнесу.

**Sellerboard робить дві речі:**
1. Рахує **Net Profit** і **Margin** після всіх Amazon fees, FBA, реклами, повернень — для кожного ASIN і для кожного портфоліо.
2. Обчислює **Break-even ACoS** — ACoS, при якому PPC рівно окупає себе з урахуванням реальних витрат.

Обидві метрики — вхідні дані для формули `target ACoS = max(Margin, BE-ACoS), cap 30, floor 10` (див. Модуль 5 Крок 1). Без Sellerboard ти їх не отримаєш.

## 2. Sellerboard Товари — структура колонок, які парсить Skill 1

**Файл:** `[ACCOUNT]_Товари_Группировать_по_Маркетплейсу_[dates].xlsx`

Формат вигрузки — одна вкладка на маркетплейс (Amazon.com, Amazon.ca). Тижневий діапазон.

**Ключові колонки, які читає Skill 1:**

| Колонка | Що це | Де використовується |
|---------|-------|---------------------|
| `ASIN` | Amazon ASIN | ID для matching з bulk, PPC Dashboard |
| `Margin` | Net Profit / Sales × 100 (вже у %) | target ACoS (max з BE-ACoS) |
| `COG` | Cost of Goods — собівартість одиниці | перевірка, чи заповнений — без нього Margin не рахується |
| `Real ACoS` | ACoS з урахуванням всіх продажів ASIN (не тільки PPC-only) | тригер #20 (ACoS vs target), контроль TACoS |
| `Sessions` | Sesssions на картку ASIN за тиждень | тригер #7 (Sessions drop) |
| `USP` (Unit Session Percentage) | CR лістингу загалом | тригер #13 (Exact CR > USP), тригер #27 (CR anomaly) |
| `Sales` | Revenue за період | верифікація (±5% з поточним тижнем) |
| `Net Profit` | Прибуток після всіх витрат | Група 1 пріоритетів Skill 1 |
| `Refund rate` | Повернення в % | Група 2 пріоритетів (>10% → лістинг/якість, не PPC) |

**Структура процесу:**
1. Вигружай Товари **на 4 тижні окремими файлами** (щоб бачити WoW тренди, а не середнє).
2. Назва файлу — **не переіменовуй**, Skill 1 парсить дати з оригінальної назви.
3. Перевіряй, що Sales зі скріншоту поточного тижня ≈ Sales з xlsx (±5%) — інакше вигрузка неповна або пошкоджена.

**Критичний сигнал:** якщо `Margin = None` / `NULL` при PPC Spend > 0 — це тригер #4 (Skill 3). У 99% випадків причина — незаповнений COG. Повідом аккаунт-менеджеру; поки COG не проставлений, PPC-оптимізація по цьому ASIN йде наосліп.

### Коли використовувати Товари

- **Щотижня при зборі target ACoS.** Це основа Skill 1.
- **При розслідуванні просадки Net Profit.** Колонки `Margin`, `Refund rate`, `Real ACoS` за 2–4 тижні дають відповідь «де джерело втрат».
- **Коли треба перевірити, чи жива картка товару** (Sessions = 0 при Spend > 0 → suppressed listing або акаунтова проблема).

## 3. PPC Dashboard по портфоліо — джерело Break-even ACoS

**Файл:** `[ACCOUNT]_Дэшборд_DashboardEntryTypePortfolios_Группировать_по_portfolio_[dates].xlsx`

Це вигрузка рівня портфоліо (не кампанії). Різниця критична: **BE-ACoS беремо ТІЛЬКИ з portfolio-level**, campaign-level BE ігноруємо, бо там формула рахує без повного урахування fees по портфоліо → значення неточне.

**Фільтри при читанні файлу:**
- `Status` = порожня клітинка (NaN) — це рядок рівня портфоліо (не кампанії).
- `Marketplace` ≠ порожнє (Amazon.com / Amazon.ca).

Ці два фільтри зменшують файл з 4000+ рядків до 50–100 — кожен рядок одне портфоліо.

**Парсинг ASIN з назви портфоліо:** regex `B0[A-Z0-9]{8}` — перший матч.

**Ключові колонки:**

| Колонка | Що це |
|---------|-------|
| `Portfolio name` | `[XX%] B0XXXXX - назва` — XX% це portfolio target ≈ Margin |
| `Break even ACOS` | Основа для target ACoS |
| `ACOS` | Реальний ACoS портфоліо за період |
| `Spend`, `Sales`, `Orders` | Агрегати для контексту |

### Коли використовувати PPC Dashboard

- **Щотижня при зборі target ACoS** (разом з Товари).
- **При перевірці групи 1** Skill 1 — портфоліо з зростанням Real ACoS >3 pp WoW або зростанням Spend >15% без росту Net Profit.
- **При неймінгу портфоліо.** `[XX%]` префікс — це portfolio target (Margin). Помічає менеджерам одразу бачити прибутковість.

## 4. Stock Planner — сток як обмежувач PPC

**Шлях:** Sellerboard → Планувальник → вибрати маркетплейс.

Ключова колонка — `Days of Stock` (днів до вичерпання).

### Правила, прив'язані до PPC

| Days of Stock | Дія по PPC |
|---------------|------------|
| `< 10` | Знизити біди на −20 до −30%, або паузити частину кампаній — інакше спалиш бюджет, а товару не буде |
| `10–30` | Обережна оптимізація, не підвищувати бюджети агресивно |
| `30–100` | Штатний режим |
| `> 100` | Можна розкатувати (підвищувати бюджет / біди) — надлишок стоку треба продавати |
| `= 0` (out of stock) | Автоматично негатив для всієї оптимізації — рекомендується повна пауза PPC до поповнення |

**Коли використовувати Stock Planner:**
- Раз на тиждень як обов'язковий чек перед будь-якими bulk-змінами.
- Перед запуском нових кампаній (нема сенсу запускати агресивну кампанію на ASIN з 8 днями стоку).
- Як ескалаційний сигнал — при стоку <10 днів повідомляй аккаунт-менеджера.

## 5. Тижневий workflow — як усе збирається разом

1. **Вигрузка Товари × 4 тижні** (формат `Группировать_по_Маркетплейсу`) — US + CA окремими файлами.
2. **Вигрузка PPC Dashboard по портфоліо** — один файл (US + CA разом).
3. **Верифікація Sales** — скріншот поточного тижня vs xlsx (±5%).
4. **Stock Planner** — зафіксувати ASIN з `Days of Stock` <10 і >100.
5. **Обчислення target ACoS** для кожного ASIN → `max(Margin, BE-ACoS), cap 30, floor 10`.
6. **Group sorting** за 5 пріоритетами Skill 1 (від Групи 1 — критичні просадки — до Групи 5).
7. **Передача в Skill 2** (bulk optimization) і Skill 3 (тригерна діагностика).

Будь-який пропущений крок означає, що оптимізація далі буде неточною.

---

## 6. Типові помилки читання Sellerboard

- **ACoS у коефіцієнтах, не у відсотках.** Якщо `max(ACoS)` у файлі < 2.0 → всі ACoS множимо на 100. Без цього `0.35 > 25` завжди False, і тригери не спрацьовують.
- **Читаєш campaign-level BE-ACoS замість portfolio-level.** Різниця в розрахунку fees — значення буде неточне, target ACoS вийде кривий.
- **Не перевіряєш Sales-верифікацію.** Неповна вигрузка (наприклад, за 6 днів замість 7) змістить усі метрики. 5% розбіжність — поріг для перевигрузки.
- **Ігноруєш Refund rate.** Якщо >10% — це Група 2, PPC не вилікує. Рекомендація — ескалація лістингу / якості.

### Практичний кейс

**Сценарій.** Ти відкрив тижневу вигрузку Sellerboard Товари для акаунту. Виявив наступні ASIN у US маркетплейсі:

| ASIN | Margin | Real ACoS | Sessions WoW | USP | Refund rate | Days of Stock | Spend/тиждень |
|------|--------|-----------|--------------|-----|-------------|---------------|----------------|
| B07AAA | 21% | 18% | −5% | 12.4% | 3% | 45 | $220 |
| B07BBB | NULL | 32% | −8% | 10.1% | 7% | 60 | $140 |
| B07CCC | 9% | 27% | +2% | 9.3% | 14% | 8 | $180 |
| B07DDD | 28% | 16% | +40% | 11.8% | 4% | 150 | $95 |

BE-ACoS по портфоліо відповідно: 15%, 17%, 19%, 12%.

**Питання:** для кожного ASIN визнач:
1. target ACoS.
2. Групу пріоритетів Skill 1.
3. Основну дію (включаючи зв'язок зі Stock Planner).

*Розв'язок:*

- **B07AAA:** target = max(21, 15) = 21%. Real ACoS 18% нижче — кампанія прибуткова. Група 4–5 (стабільна). Дія: штатний тижневий контроль, можна розглянути структурні тригери Модуль 6 (#17, #19) якщо мало кампаній.

- **B07BBB:** Margin=NULL при Spend=$140 → **Група 1, тригер #4**. Target не рахуємо — нема даних. Дія: зупинити всяку оптимізацію по ASIN, повідомити аккаунт-менеджеру про відсутній COG.

- **B07CCC:** target = max(9, 19) = 19%. Real ACoS 27% перевищує на +8 pp. АЛЕ критичніше — Refund rate 14% (Група 2) і Days of Stock 8 (критичний сток). Пріоритет дій: знизити біди −25% через низький сток; НЕ інвестувати час у PPC-оптимізацію по причині повернень (треба лістинг/якість через менеджера).

- **B07DDD:** target = max(28, 12) = 28% → cap 30%, залишається 28%. Real ACoS 16% — суттєво нижче target, ASIN прибутковий і росте (+40% Sessions, Days of Stock 150 — надлишок). Група 4. Дія: це кандидат на **розкатку** — підвищити бюджети, розглянути тригери #11 (bottleneck), #17 (нема SB/SD — запустити), #19 якщо кампаній мало.

### Розрахункова задача

Дано рядок PPC Dashboard по портфоліо для ASIN B07TEST у Amazon.com:

- Portfolio name: `[18%] B07TEST - Spice Jars`
- Status: (порожній)
- Marketplace: Amazon.com
- Sales (portfolio): $4,820
- Spend (portfolio): $1,156
- ACOS (portfolio): 0.24
- Break even ACOS: 0.13

Sellerboard Товари для B07TEST: Margin = 18%, Real ACoS = 23%, Sessions = 1,420, USP = 9.8%, Refund rate = 4%, Days of Stock = 55.

**Порахуйте:**
1. Optimization target ACoS (формула Модуль 5 Крок 1).
2. ACoS портфоліо у відсотках (перевір формат).
3. Наскільки фактичний ACoS перевищує target (pp і разів).
4. Яка група пріоритетів Skill 1 і чому?
5. Який перший тригер з Skill 3 тут спрацьовує?

*Розв'язок:*
1. target = max(Margin 18, BE 13) = **18%**, cap 30 не спрацьовує, floor 10 не потрібен.
2. ACoS у файлі = 0.24 — формат коефіцієнтів (< 2.0 правило). Real ACoS у %: **24%**. BE-ACoS = **13%**.
3. 24% − 18% = **+6 pp**, 24 / 18 ≈ **1.33× target**.
4. Група 1 — Real ACoS >3 pp вище target на стабільному ASIN при Spend >$1000 (+6 pp = тригер ACoS зростання). Margin не просіла критично, але кампанія `працює у мінус` відносно профіту (платимо більше, ніж допускає маржа).
5. **Тригер #20 (ACoS > target)** — основний. Рекомендація: поетапне зниження бідів (−10 до −15% на exact/phrase) + перевірка структурних тригерів (#16, #17, #19) для розвантаження уваги на слабкі кампанії.
